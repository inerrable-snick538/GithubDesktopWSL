## ⚙️ How It Works

GithubDesktopWSL connects GitHub Desktop to files in WSL. That means Git can work closer to where your files live instead of going through slower shared file paths. For many projects, this cuts wait time when you open a repo, check status, or switch branches.

You do not need to change how you use your files every day. You can keep working in Windows while your code stays in WSL.

## 🛠️ Setup Tips

- Keep your repo inside the WSL file system for best results.
- Avoid storing WSL repos inside Windows shared folders.
- Use one repo path for each project to reduce confusion.
- If the app asks to refresh a repo, let it finish before making more changes.
- Close extra file tools if the repo feels slow.

## 🔍 Troubleshooting

### App does not open
- Restart Windows.
- Open the app again from the Start menu.
- Make sure the install finished without errors.

### WSL repo does not appear
- Confirm that WSL is installed.
- Check that the repository is inside your WSL file system.
- Refresh the repository list.

### Git feels slow
- Use a repo stored in WSL2.
- Close other apps that scan the same folder.
- Check that Windows Defender is not scanning the repo too often.

### Changes do not show up
- Save your files in your editor.
- Wait a moment for file sync.
- Refresh the repo view.

## 📁 File and Folder Basics

If you are new to WSL, think of it like a Linux space inside Windows. Your project files should stay in that space when you want the best speed. GitHub Desktop then reads those files with less delay.

A good rule is simple:
- WSL project files in WSL
- Windows project files in Windows

Mixing the two can make things harder to follow.
